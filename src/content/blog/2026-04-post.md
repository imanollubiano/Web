## Introducción

La gestión centralizada de contraseñas y secretos es un pilar fundamental en infraestructuras modernas. Después de una década trabajando con sistemas en producción, he visto cómo el almacenamiento descentralizado de credenciales causa brechas de seguridad, cumplimiento deficiente y auditorías fallidas. Este artículo aborda soluciones probadas: Passbolt para equipos colaborativos y HashiCorp Vault para orquestación compleja de secretos.

## El problema real en producción

En entornos empresariales típicos, las credenciales se dispersan en:
- Archivos `.env` comprometidos en repositorios Git
- Documentos compartidos en Confluence sin control de acceso
- Scripts legacy con contraseñas hardcodeadas
- Notas personales de administradores

Esta fragmentación genera:
- **Compliance gaps**: Imposible auditar quién accedió a qué credencial
- **Credential sprawl**: Múltiples copias de la misma credencial sin sincronización
- **Rotación manual**: Cambios manuales de contraseñas sin automatización
- **Revocación lenta**: Días o semanas para revocar acceso tras un incidente

## Passbolt: Gestión colaborativa de contraseñas

Passbolt es una solución de código abierto diseñada para equipos que necesitan compartir contraseñas con control granular. A diferencia de gestores personales (1Password, LastPass), Passbolt está orientado a sysadmins y DevOps.

### Arquitectura y seguridad

Passbolt utiliza encriptación OpenPGP de extremo a extremo. Las contraseñas se encriptan en el cliente antes de transmitirse al servidor, lo que significa que el servidor nunca almacena secretos en texto plano.

```yaml
# Configuración básica de Passbolt en Docker
version: '3.8'
services:
  passbolt:
    image: passbolt/passbolt:latest-ce
    environment:
      PASSBOLT_FLAVOUR: ce
      PASSBOLT_SSL_FORCE: "true"
      PASSBOLT_REGISTRATION_PUBLIC: "false"
      PASSBOLT_DB_DEFAULT_HOST: db
      PASSBOLT_DB_DEFAULT_USERNAME: passbolt
      PASSBOLT_DB_DEFAULT_PASSWORD: "${DB_PASS}"
    volumes:
      - ./config/gpg:/etc/passbolt/gpg
      - ./config/jwt:/etc/passbolt/jwt
    depends_on:
      - db
    ports:
      - "443:443"
  
  db:
    image: mariadb:10.6
    environment:
      MARIADB_ROOT_PASSWORD: "${ROOT_DB_PASS}"
      MARIADB_DATABASE: passbolt
      MARIADB_USER: passbolt
      MARIADB_PASSWORD: "${DB_PASS}"
    volumes:
      - db_data:/var/lib/mysql

volumes:
  db_data:
```

### Casos prácticos de Passbolt

**Caso 1**: Equipo de 15 sysadmins que necesita acceso compartido a credenciales de MySQL, Redis y balanceadores de carga. Passbolt permite:
- Crear grupos de acceso (Prod DBA, Networking, Security)
- Auditar quién accedió a cada credencial con timestamps
- Revocar acceso inmediatamente sin cambiar la contraseña
- Integración con navegadores mediante plugin

**Configuración de acceso granular**:
```bash
# CLI de Passbolt para automatización
passbolt_cli create-secret \
  --name "MySQL Prod Primary" \
  --username "admin_prod" \
  --password "$(openssl rand -base64 32)" \
  --description "Master DB 192.168.1.50" \
  --uri "mysql://192.168.1.50:3306"

# Asignar a grupo específico
passbolt_cli add-permission \
  --resource-id "resource-uuid" \
  --group-id "group-prod-dba" \
  --permission "read"
```

## HashiCorp Vault: Orquestación de secretos en escala

Vault es la opción para infraestructuras complejas que requieren:
- Rotación automática de credenciales
- Tokens de corta duración (TTL dinámico)
- Integración con aplicaciones y CI/CD
- Múltiples backends de almacenamiento

### Arquitectura de Vault

Vault opera con tres componentes clave:
1. **Storage Backend**: Dónde se almacenan los secretos (Consul, S3, etcd)
2. **Auth Methods**: Cómo se autentica (Kubernetes, LDAP, AWS IAM, tokens)
3. **Secret Engines**: Cómo se generan/rotan (Database, PKI, SSH, Dynamic Secrets)

```hcl
# Configuración de Vault para rotación automática de MySQL
vault {
  address = "https://vault.prod.internal:8200"
  retry {
    num_retries = 3
  }
}

# Definir conexión a base de datos
vault write database/config/mysql \
  plugin_name=mysql-database-plugin \
  allowed_roles="readonly,admin" \
  connection_url="{{username}}:{{password}}@tcp(mysql.prod:3306)/" \
  username="vault_rotation" \
  password="initial_password"

# Crear rol con rotación automática (cada 24 horas)
vault write database/roles/readonly \
  db_name=mysql \
  creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}'; GRANT SELECT ON *.* TO '{{name}}'@'%';" \
  default_ttl="1h" \
  max_ttl="24h"

# Desde la aplicación: obtener credencial temporal
vault kv get database/readonly
# Retorna: user_readonly_randomsuffix con password que expira en 1h
```

### Integración con Kubernetes

En entornos Kubernetes, Vault se integra mediante Kubernetes Auth Method:

```yaml
# Configuración de pod que consume secretos de Vault
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: production
---
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: myapp:latest
    env:
    - name: VAULT_ADDR
      value: "https://vault.internal:8200"
    - name: VAULT_ROLE
      value: "app-role"
    volumeMounts:
    - name: jwt
      mountPath: /var/run/secrets/kubernetes.io/serviceaccount
  volumes:
  - name: jwt
    projected:
      sources:
      - serviceAccountToken:
          audience: vault
          expirationSeconds: 600
          path: token
```
